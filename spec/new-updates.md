# Spec: Simplified Blog Publishing Pipeline

**Version:** 1.1  
**Date:** 2026-06-20  
**Status:** Approved

**Changelog v1.1:** Removed TL;DR blocks from enrichment rules (caused unstyled text on live posts). Added explicit rules prohibiting H1 and image markers in `content.md` — both were causing duplication on the live page because the theme renders them independently from the post's `title` and `featured_media` fields. Added wp-publisher pre-conversion strip rules as a safety net.

---

## 1. Problem Statement

The original pipeline had too many steps — spec generation, human approval gate, separate image-creator and diagram-creator agents, and a multi-command workflow. This created friction and slowed down publishing.

The new design reduces the pipeline to a single user action: write content in `blog_content.md`, run `/blog`, and the system handles enrichment, SEO, image selection, and WordPress publishing automatically.

---

## 2. New Pipeline Overview

```
User writes → blog_content.md
                    ↓
           [/blog command]
                    ↓
        seo-writer agent
        - Reads blog_content.md
        - Enriches content (adds missing sections/headlines)
        - Enforces no-emoji rule
        - Runs SEO pass (derives keyword, slug, meta)
        - Writes: drafts/<slug>/content.md
                  drafts/<slug>/metadata.json
                    ↓
        Image selection
        - Reads images_names skill catalog
        - Picks best-matching hero image (1280×720)
        - Source: drafts/images_to_use/images/pre_images/AI_Blog_1280_720/
        - Copies to: drafts/<slug>/images/<slug>-featured.png
        - Writes: drafts/<slug>/images/manifest.json
                    ↓
        wp-publisher agent
        - Uploads featured image to WP Media Library
        - Converts content.md to Gutenberg HTML
        - Creates WP post (draft by default)
        - Writes: drafts/<slug>/publish-log.json
                    ↓
        Reports: post ID, admin edit URL, preview URL
```

**Single command:** `/blog` (no flags required for basic use)  
**Live publish:** `/blog --live` or use `/publish <slug> --live` separately  
**No spec step. No human approval gate before writing.**

## 3. Input Contract

### 3.1 Primary Input: `blog_content.md`

- Located at the **project root**: `/blog_content.md`
- User writes raw blog content here — any format is accepted (rough outline, full draft, bullet points)
- The file **must not be empty** before running `/blog`
- No emoji should be used by the user (but the agent will strip any that appear regardless)

**Minimum viable content in `blog_content.md`:**
- A title or subject line (first line or H1)
- At least 3-5 paragraphs or key points the blog should cover

**What the user does NOT need to provide:**
- SEO keywords (derived by the agent)
- Slug (derived from title)
- Meta description (generated by agent)
- Image selection (handled automatically)
- WordPress credentials (read from `.env`)

### 3.2 Environment Variables (`.env`)

Required:
```
WP_BASE_URL=https://www.tecorb.com
WP_API_BASE=https://www.tecorb.com/wp-json/wp/v2
WP_USER=<editor-username>
WP_APP_PASS=<application-password>
WP_DEFAULT_AUTHOR_ID=<numeric-id>
```

Optional:
```
INDEXNOW_KEY=<key>   # only used on live publish
```

## 4. Content Enrichment Rules

The seo-writer agent must follow these rules when enriching `blog_content.md`:

### 4.1 No Emoji — Hard Rule
- Strip all emoji from user-written content
- Never introduce emoji in generated content
- This applies to: body text, headings, callout boxes, CTAs, meta description, slug, everything

### 4.2 Content Additions
The agent reviews the user's content and adds:
- A strong opening paragraph if the user's intro is weak or missing
- Missing H2 sections that a reader would reasonably expect, based on the topic
- A FAQ section (5-7 questions) if one is absent and the post is informational
- A closing CTA aligned to Tecorb's services

**Never add a TL;DR block.** The theme has no styling for it — the label renders as unstyled bold text ("TL;DR") on the live page.

### 4.3 What the Agent Must NOT Change
- The user's core factual claims
- The user's chosen examples or named references
- The overall intent and subject of the post
- Any statistics or numbers the user provided (flag them but do not alter)

### 4.4 content.md Structural Rules — Hard

Two elements must never appear in `content.md` because the WordPress theme renders them independently, causing duplication on the live page:

| Element | Why it duplicates | Rule |
|---|---|---|
| **H1 / post title** | Theme renders the title from the post's `title` field above all content | Do NOT write `# Title` at the top of content.md |
| **Featured image** | Theme renders the featured image from the `featured_media` field before the content | Do NOT add `[IMAGE: featured]` or any image marker |

`content.md` must start directly with the first body paragraph or the first H2 section.

### 4.5 Headline Quality
Every H2 and H3 must:
- Be specific and keyword-bearing (not generic like "Introduction" or "Conclusion")
- Match the Tecorb style guide heading patterns
- Be free of emoji

### 4.6 Tecorb Style Compliance
- Follow `skills/tecorb-style-guide/SKILL.md` for tone, vocabulary, callout box syntax, and CTA placement
- Follow `skills/seo-blog-writing/SKILL.md` for keyword density, snippet optimisation, and E-E-A-T signals
- No banned phrases from the style guide vocabulary section

## 5. SEO Rules

The seo-writer agent derives all SEO fields from the content — no user input required.

### 5.1 Fields Generated
| Field | Rule |
|---|---|
| `focus_keyword` | Primary search intent phrase, extracted from title + main body theme |
| `slug` | Kebab-case from title, stop words removed, max 60 chars |
| `seo_title` | H1 + brand suffix if needed, max 60 chars |
| `meta_description` | 130-155 chars, includes focus keyword, describes the post value |
| `secondary_keywords` | 3-5 related terms from body content |
| `category_slugs` | Derived from topic category (e.g., `ai-ml`, `mobile`, `devops`) |
| `tag_slugs` | Specific technologies named in the post |

### 5.2 SERP Check
Before finalising the outline, the agent searches the focus keyword and checks the top 5 results to:
- Confirm the detected search intent matches the content
- Identify gaps the existing post can fill
- Validate or adjust the heading structure

## 6. Image Selection Rules

### 6.1 Hero Image Only
Each blog post gets exactly **one hero image** (1280×720). No in-body images are used in this pipeline.

### 6.2 Source Folder
```
drafts/images_to_use/images/pre_images/AI_Blog_1280_720/
```
This is the only folder used. The `AI_Blog_450_450/` folder is not used in this pipeline.

### 6.3 Selection Process
1. Read the `images_names` skill — consult the Summary Reference Table and Image about descriptions
2. Match the blog's primary topic to the image category and key technologies
3. Select the single best-matching hero image
4. Verify the file exists in the source folder before copying
5. Copy the selected file to `drafts/<slug>/images/<slug>-featured.png`
6. Write `images/manifest.json` with the selection

### 6.4 Manifest Format (featured image only)
```json
{
  "featured": {
    "source_file": "1. Custom LLM & SLM Development_Hero_Page.png",
    "source_folder": "drafts/images_to_use/images/pre_images/AI_Blog_1280_720",
    "local_copy": "drafts/<slug>/images/<slug>-featured.png",
    "alt": "Cinematic illustration of a neural network being engineered inside a holographic workspace.",
    "caption": null,
    "role": "featured"
  }
}
```

### 6.5 Alt Text Rule
- 8-15 words describing what is visually depicted
- No emoji, no keyword stuffing
- Describes the image, not the blog topic

## 7. Output Structure

A completed run produces:

```
drafts/<slug>/
├── content.md           ← enriched blog content, no emoji
├── metadata.json        ← SEO fields, categories, tags, schema blocks
└── images/
    ├── manifest.json    ← featured image entry only
    └── <slug>-featured.png   ← copied from AI_Blog_1280_720
```

**No spec.json. No spec.md. No diagrams folder.**

### 7.1 metadata.json Schema
```json
{
  "title": "Post H1 title",
  "seo_title": "SEO title tag (max 60 chars)",
  "slug": "url-slug",
  "excerpt": "WordPress excerpt (max 155 chars)",
  "meta_description": "Meta description (130-155 chars)",
  "focus_keyword": "primary keyword",
  "secondary_keywords": ["kw1", "kw2", "kw3"],
  "category_slugs": ["ai-ml"],
  "tag_slugs": ["llm", "fine-tuning"],
  "schema_blocks": [
    {"type": "Article"},
    {"type": "FAQPage", "questions": []}
  ],
  "author_id": null,
  "word_count": 0,
  "reading_time_min": 0
}
```

Note: `diagram_specs` and `image_specs` fields are removed in this pipeline version.

## 8. WordPress Publish Contract

### 8.1 Default Mode: Draft
Unless `--live` is explicitly passed, the wp-publisher always creates the post as WP draft status. The user reviews in WP admin and publishes manually (or runs `/publish <slug> --live`).

### 8.2 Live Publish
Requires explicit flag: `/blog --live` or `/publish <slug> --live`.  
Before publishing live, the agent asks: "About to publish live to tecorb.com. Reply yes to continue."  
Only `yes`, `y`, or `confirm` proceed. All other responses abort.

### 8.3 Post Creation Steps
1. Verify WP auth (`GET /wp-json/wp/v2/users/me` → must return 200)
2. Check slug is not already taken
3. Upload featured image to WP Media Library → record `featured_media` ID
4. Pre-process `content.md` — strip the following before Gutenberg conversion:
   - Any top-level H1 (`# ...`) — theme renders title from the post's `title` field
   - Any `[IMAGE: ...]` markers — theme renders featured image from `featured_media` field
   - Any TL;DR block (`**TL;DR**` label + following bullet list) — theme has no styling for it
5. Convert cleaned content to Gutenberg HTML
6. Inject JSON-LD schema blocks
7. Resolve or create categories and tags
8. Create post via `POST /wp-json/wp/v2/posts`
9. Verify SEO meta fields saved correctly
10. If live mode: submit URL to IndexNow
11. Write `publish-log.json`

### 8.4 publish-log.json
```json
{
  "run_at": "<ISO datetime>",
  "mode": "draft|publish",
  "post_id": 12345,
  "post_url": "https://www.tecorb.com/...",
  "admin_edit_url": "https://www.tecorb.com/wp-admin/post.php?post=12345&action=edit",
  "featured_media_id": 4567,
  "indexnow_submitted": false,
  "warnings": []
}
```

## 9. Commands

### `/blog [--live]`
The primary command. Reads `blog_content.md` from the project root and runs the full pipeline end-to-end.

| Flag | Effect |
|---|---|
| *(no flags)* | Produces a WP draft |
| `--live` | Publishes immediately (requires `yes` confirmation) |

### `/publish <slug> [--live] [--schedule=ISO]`
Re-publishes or transitions an existing draft. Used when:
- The user wants to publish a previously created WP draft
- Scheduling a post for future publication
- Updating an existing post (`--update=<id>` flag)

## 11. Constraints and Guard Rails

- **Never generate emoji** anywhere in the output — content.md, metadata.json, headings, CTAs, nothing
- **Never invent statistics** — if the user didn't provide a number, do not fabricate one
- **Never auto-publish live** without explicit `--live` flag + `yes` confirmation
- **Never overwrite an existing WP post** without `--update=<id>` and user confirmation
- **Never commit `.env`** — the gitignore rule must stay in place
- **If `blog_content.md` is empty**, stop immediately and tell the user to add content
- **If WP auth fails** (401/403), stop and tell the user to rotate the Application Password
- **If the slug is already taken** in WordPress, ask the user before proceeding

## 12. Verification Checklist

After a `/blog` run, verify:

- [ ] `drafts/<slug>/content.md` exists and contains no emoji
- [ ] `content.md` does NOT start with an H1 heading
- [ ] `content.md` contains no `[IMAGE: ...]` markers
- [ ] `content.md` contains no TL;DR block
- [ ] `drafts/<slug>/metadata.json` has valid `slug`, `seo_title`, `meta_description`, `focus_keyword`
- [ ] `drafts/<slug>/images/<slug>-featured.png` copied from `AI_Blog_1280_720/`
- [ ] `drafts/<slug>/images/manifest.json` has a `featured` entry with `alt` text
- [ ] WP post created — `publish-log.json` contains a valid `post_id`
- [ ] Admin edit URL opens the post in WP admin — title appears once, featured image appears once
- [ ] SEO meta fields visible in WP post edit view (`context=edit` API check)
